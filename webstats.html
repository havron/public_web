Magic used to deliver 'havron.xyz'

This site is developed and maintained by myself, and utilizes a number of recent web tools and standards to operate. The majority of my knowledge of web development is self-taught, though I am taking a course on building scalable, industrial-grade 4-tier web applications during fall 2016. I'm always looking to improve the site or add new content! Feel free to contact me or submit a pull request.

Dev tools and infrastructure

Most of the site development is done with the help of Vim and Google Chrome (with the use of many DuckDuckGo bangs!). Deployment is handled with an AWS Cloudfront CDN and S3 origin; Route 53 takes care of the 'havron.xyz' domain, and AWS ACM serves as the Certificate Authority (CA) for the site's TLS certificate and HTTPS connection. Styling is based on Twitter bootstrap (specifically, the Cyborg theme on Bootswatch). Lastly, version control with git is used and publicly available in a GitHub repository (the site's S3 bucket is synchronized with the repository via a client-side hook).

Site blog

The blog is developed using Hugo to generate static content. It's currently housed in the /blog subdirectory of the site, but could migrate to its own subdomain or entirely separate domain if it ever becomes big enough to invite multiple contributors whom would need access to a shared, distinct blog environment. AWS offers the granularity to control S3 and Cloudfront resources at the directory level, which could certainly come in handy in the aforementioned scenario.

The blog's commenting system is currently housed on reddit, whom handles all of the dynamic needs of blog comments. Each reddit post corresponds to a blog post, and is embedded in JavaScript at the bottom of the relevant blog post. I may turn to implementing my own commenting system in the future, but will stick with reddit for now. I am open to suggestions for a system that you (the reader) would prefer to use; fill out this form! The blog's theme is based on Bootswatch.

Security

The site is static, which is a great start towards ensuring it is safe and secure for viewers to browse. As a quick curl -I HTTP-header request can reveal, The site relies on a 301 redirect from an initial HTTP request to HTTPS connection (which invites some flaws), enforced by the Cloudfront distribution that serves the site's objects from an S3 bucket with match viewer protocol policy (so, Cloudfront will establish an HTTPS connection to the S3 bucket and serve content via HTTPS to the viewer). I would rather have HSTS (HTTP Strict Transport Security) headers and an HTTPS-only connecton enforced (cf. cookie integrity problems), but HSTS headers are currently ignored in AWS S3 buckets and in AWS Cloudfront, both as of June 2016; the site's only cookies are for a Google Analytics script, which could potentially be poisoned by a MITM (Man in the Middle) attack during the initial HTTP request to the Cloudfront distribution. Also as a result of using Amazon Certificate Manager as a CA, Perfect Forward Secrecy (PFS) is not supported, which isn't much of a concern for a static site, but is still on my mind.

Search Engine Optimization (SEO)

While I profess that SEO is not at the top of my priority list for the site, I have had quite a bit of interest in learning more about the algorithms behind popular search engines; what better way to learn than to experiment with my own site? I haven't yet taken a stance on using absolute URLs (supposedly better for SEO) over relative URLs (to me, much more intuitive and efficient for development). More on this section to be added.
